# 13장 과제 – 웹 애플리케이션과 영속성 관리

## 공부한 내용 요약
- J2SE vs J2EE
    - standard edition vs enterprise edition
    - jvm vs was

- 트랜잭션 범위의 영속성 컨텍스트
    - 트랜잭션 범위 = 영속성 컨텍스트의 생존 범위
    - 트랜잭션 실패 시, 플러쉬를 안하기 때문에 주의 필요.
    - 트랜잭션이 같으면 같은 영속성 컨텍스트를 사용한다.
    - 스프링 컨테이너는 스레드 마다 다른 트랜잭션을 할당 ⇒ 같은 em을 호출해도 다른 tx이 할당되기 때문에 멀티 스레드 상황에서 안전하다.

- view 단에서는 준영속 상태가 됨
    - 그럼 엔티티의 지연로딩을 사용할 수가 없음.
        - 글로벌 페치 전략 = 즉시 로딩을 사용함.
            - 이는 N+1 문제 발생
        - 페치 조인 : 현식적인 대안.

- N + 1 문제
    - JPQL을 사용할 때 발생함.
    - JPA는 JPQL 분석해서 쿼리 생성 시에는 글로벌 페치 전략을 참고하지 않는다.
    - Order 10개 조회 시 각 order의 Member를 조회하는 쿼리도 10번 추가로 발생됨.

- OSIV : 영속성 컨텍스트를 뷰 계층까지 열어두자
    - 요청 당 트랜잭션 방식 : 요청 왔을때부터 TX 열자 → 뷰 계층에서 엔티티를 수정할 수 있다는 단점
        - 단점 : 같은 영속성 컨텍스트를 여러 TX에서 사용 가능 (전통적 OSIV에서 요청 안에서 여러 TX가 실행된다는 의미임)
    - 스프링 OSIV : 영속 상태이지만 수정 불가능하도록 지원. (뷰 계층에서 TX를 없앴기 때문에 조회만 가능)
        - 주의 : 뷰에서 변경 후 비즈니스 로직 태우면 변경됨.


## 5) 과제 하며 생각해볼 것들 -> 발표자가 이야기 해보아요!

#### Scenario A : 생각해볼 것
- A-1은 `save()`를 안 불렀는데 **DB에 반영되는가?** => 반영됨
- A-2는 동일한 로직인데 **DB에 반영되지 않는가?** => 반영 안됨
- 두 결과가 다른 이유는? => @Transactional이 영속성 컨텍스트의 생명주기를 결정하기 때문. Transactional 종료 시 flush 발생됨.

#### Scenario B : 생각해볼 것
- B-1에서 반환된 Member의 `getTeam().getName()`을 호출하면 **`LazyInitializationException`이 발생하는가?** => 발생함
- B-2에서는 같은 호출이 **정상 동작하는가?** => 정상 동작함
- 왜 차이가 나는가? =>
  - B-1은 트랜잭션 종료 후 지연 로딩 시도 → 영속성 컨텍스트가 이미 종료되어 LazyInitializationException 발생. 
  - B-2는 fetch join으로 트랜잭션 안에서 Team까지 함께 조회 → 이미 조회되어 있기 때문에 LazyInitializationException 발생하지 않음.

### (보너스) Scenario C: OSIV로 해결하기

`Chapter13BonusTest`를 실행하면, OSIV를 `true`로 오버라이드한 상태에서 B-1의 `findMember()`를 다시 호출합니다.
이번에는 `LazyInitializationException`이 **발생하지 않습니다**. 왜 그런지 생각해보세요.

- OSIV=true일 때 영속성 컨텍스트가 어디까지 살아있는가? => API 응답이 끝날 때까지 살아있음
- OSIV=true가 편리하지만 실무에서 끄는 이유는? => 성능 문제 추적 어렵다고 함 (책에 나와있음)

#### Q1. @Transactional이 영속성 컨텍스트의 생명주기를 결정한다는 것은 무슨 의미인가?
- 트랜잭션 시작 = 영속성 컨텍스트 생성
- 트랜잭션 커밋 = flush + 영속성 컨텍스트 종료
- 이 범위 밖에서 엔티티는 어떤 상태가 되는가? => 준영속 상태 (지연 로딩, 변경 감지 불가)

#### Q2. 변경 감지 vs merge(), 실무에서는 어떤 것을 써야 하나?
- `merge()`의 위험성: 모든 필드를 덮어씌운다는 것은 무슨 의미인가?
- 예: 이름만 바꾸려고 `merge()`를 호출했는데 다른 필드가 null로 덮어씌워진다면? => 큰 장애로 번질 수 있음. merge 사용 시 주의 필요.

#### Q3. fetch join 말고 다른 해결 방법은 없나?
- `@EntityGraph` 활용
- DTO로 직접 조회 (`new` 키워드 사용 JPQL)
- 각각의 장단점은? =>
  - fetch join은 간단하지만, 복잡한 쿼리에서는 가독성이 떨어질 수 있음.
  - EntityGraph는 재사용 가능하지만, JPQL보다 유연성이 떨어질 수 있음.
    - @EntityGraph(attributePaths = ["team"])
      fun findById(id: Long): Member?
  - DTO는 가장 명확하지만, 유지보수 비용이 높을 수 있음. (무한 DTO 생성의 굴레)

#### Q4. OSIV=true가 편리한데 실무에서는 왜 끄는가?
- OSIV=true이면 영속성 컨텍스트가 API 응답이 끝날 때까지 살아있다 — DB 커넥션도 그때까지 반환되지 않는다면? => 커넥션 풀이 고갈, 서비스 장애로 이어짐.
- 트래픽이 많은 서비스에서 커넥션 풀이 고갈되면 어떤 일이 벌어지는가? => (많이 봤었죠..) 신규 요청이 커넥션 못얻음 => 타임아웃 => 서비스 장애
- 컨트롤러/뷰에서 의도치 않게 지연 로딩 쿼리가 나가면 성능 문제를 추적하기 어려운 이유는? => N + 1 문제 등 지연 로딩이 어디서 발생했는지 추적하기가 어려움.

#### Q5. 반대로, OSIV=true가 적합한 경우는?
- 트래픽이 적은 어드민/백오피스 시스템이라면? => 적합
- 빠르게 프로토타이핑해야 하는 초기 단계라면? => 적합
- OSIV를 끄면 모든 지연 로딩을 fetch join이나 DTO로 해결해야 하는데, 그 비용 대비 효과가 항상 맞는가? => 
  - 상황에 따라 다름. 트래픽이 적고, 개발 속도가 중요한 초기 단계에서는 OSIV=true가 편리할 수 있음. 하지만 장기적으로는 OSIV=false로 가는 것이 좋음.
