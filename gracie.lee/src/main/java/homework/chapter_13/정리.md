# 13장: 웹 애플리케이션과 영속성 관리

## 핵심 1: @Transactional = 영속성 컨텍스트의 생명주기
```
@Transactional 시작 → EntityManager 생성 → 영속성 컨텍스트 생성
  (이 안에서: 1차 캐시, 변경 감지, 지연 로딩 전부 동작)
@Transactional 종료 → flush → 커밋 → 영속성 컨텍스트 종료
  (밖에서: 준영속 상태 → 변경 감지 X, 지연 로딩 X)
```

## 핵심 2: 변경 감지 (Dirty Checking)
- `@Transactional` 안에서 엔티티 값을 바꾸면 **save() 없이도** DB에 반영됨
- `@Transactional` 없으면 같은 코드라도 DB에 반영 안 됨
- 실무에서는 `merge()` 대신 변경 감지 사용 (merge는 모든 필드를 덮어써서 위험)

## 핵심 3: LazyInitializationException
- 트랜잭션 끝나면 영속성 컨텍스트도 끝남 → LAZY 프록시 초기화 불가 → 예외 발생
- **해결법**: fetch join으로 필요한 연관 엔티티를 미리 함께 조회

## 핵심 4: OSIV (Open Session In View)
- `true` (기본값): 영속성 컨텍스트가 HTTP 요청 끝까지 살아있음 → 컨트롤러에서도 지연 로딩 가능
- `false` (실무 권장): 트랜잭션 끝나면 영속성 컨텍스트 종료
- **실무에서 끄는 이유**: DB 커넥션을 뷰 렌더링까지 물고 있으면 커넥션 풀 고갈 위험
