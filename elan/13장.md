## 13장 웹 애플리케이션과 영속성 관리

### 13.1 트랜잭션 범위의 영속성 컨텍스트

**스프링 컨테이너의 기본 전략**

- **트랜잭션 범위의 영속성 컨텍스트 전략**: 트랜잭션 시작 시 영속성 컨텍스트 생성, 종료 시 함께 종료
- 같은 트랜잭션 내에서는 항상 동일한 영속성 컨텍스트에 접근
- 여러 스레드에서 동시 요청 시 각 스레드는 서로 다른 영속성 컨텍스트 사용

**트랜잭션 생명주기**
1. 트랜잭션 시작 → 영속성 컨텍스트 생성
2. 서비스 계층에서 엔티티 조회/수정
3. 트랜잭션 커밋 → 플러시(변경 내용 DB 반영) → DB 트랜잭션 커밋
4. 예외 발생 시 → 롤백 (플러시 호출 안 함)

### 13.2 준영속 상태와 지연 로딩

**계층별 영속성 상태**

- **서비스/리포지토리 계층**: 영속 상태 유지
- **컨트롤러/뷰 계층**: 준영속 상태 (트랜잭션 종료)
- **문제점**: 준영속 상태에서 변경감지, 지연로딩 동작 안 함

**준영속 상태와 변경 감지**

- 변경 감지는 서비스 계층까지만 동작
- 프리젠테이션 계층에서 변경 감지가 동작하면 계층 책임 모호
- 명확한 책임 분리의 이점

**지연 로딩 문제**

- 영속성 컨텍스트 종료 후 지연 로딩된 프록시 객체 접근 불가
- LazyInitializationException 발생

### 13.3 OSIV (Open Session In View)

**개념**
- 요청 시작부터 응답 완료까지 영속성 컨텍스트를 열어두는 패턴

**장점**
- 뷰에서도 지연 로딩 사용 가능
- 프리젠테이션 계층에서 프록시 초기화 용이

**단점**
- 프리젠테이션 계층이 비즈니스 로직에 영향 가능
- 예기치 않은 쿼리 실행 가능
- 성능 문제 발생 가능

**OSIV 패턴 흐름**
1. 클라이언트 요청 → 영속성 컨텍스트 생성
2. 서비스 계층 호출 → 트랜잭션 시작
3. 비즈니스 로직 실행 후 → 트랜잭션 커밋, 플러시 (컨텍스트는 유지)
4. 컨트롤러/뷰 계층에서도 영속성 컨텍스트 활용
5. 요청 종료 → 영속성 컨텍스트 종료

### 13.4 너무 엄격한 계층

**문제점**
- 프리젠테이션 계층과 도메인 모델 계층의 논리적 의존성 분리 어려움
- 엔티티 직접 노출 시 보안 및 설계 문제 발생

**해결 방안**
- DTO(Data Transfer Object) 사용
- 프리젠테이션 계층에 필요한 데이터만 변환하여 전달

### 13.5 정리

**핵심 정리**

- 스프링 컨테이너는 기본적으로 트랜잭션 범위의 영속성 컨텍스트 전략 사용
- 계층별 책임을 명확히 하면서 효율적인 JPA 활용 필요
- OSIV 등의 패턴은 장단점을 고려하여 신중히 선택
- 엔티티와 DTO의 역할 분리로 웹 애플리케이션 안정성 향상
