# 13장 과제 – 웹 애플리케이션과 영속성 관리

### 과제명
**@Transactional 범위에 따른 영속성 컨텍스트 생존 범위 이해하기**

---

### 왜 이 과제를 하는가?

실무에서 JPA를 쓸 때 가장 자주 마주치는 문제 두 가지:
1. **"save() 안 불렀는데 DB가 바뀌었다"** — `@Transactional` 안의 변경 감지를 모름
2. **"아까는 됐는데 여기서는 LazyInitializationException이 터진다"** — 트랜잭션 범위 밖의 지연 로딩을 모름

이 두 문제는 **`@Transactional`이 영속성 컨텍스트의 생존 범위를 결정한다**는 것을 이해하면 해결됩니다.
이 과제는 두 상황을 직접 체험하고, 왜 그런지 설명할 수 있게 되는 것이 목적입니다.

### 목표
- `@Transactional` 범위 안/밖에서 영속성 컨텍스트의 동작 차이를 직접 체험한다.
- 변경 감지(Dirty Checking)가 동작하는 조건을 정확히 이해한다.
- 트랜잭션 밖에서 지연 로딩 시 `LazyInitializationException`이 발생하는 이유를 이해하고 해결한다.

## 5) 과제 하며 생각해볼 것들 -> 발표자가 이야기 해보아요!

#### Q1. @Transactional이 영속성 컨텍스트의 생명주기를 결정한다는 것은 무슨 의미인가?
- 트랜잭션 시작 = 영속성 컨텍스트 생성
- 트랜잭션 커밋 = flush + 영속성 컨텍스트 종료
- 이 범위 밖에서 엔티티는 어떤 상태가 되는가?
	- 트랜젝션이 시작할 때 영속성 컨텍스트를 생성 / 트랜젝션이 끝날 때 영속성 컨텍스트를 종료한다는 의미. 따라서 이 범위 밖에선 영속성 컨텍스트의 라이프사이클 밖이기 때문에 엔티티는 준영속상태이다.
	- 피드백: 준영속 상태가 되면 구체적으로 두 가지가 동작하지 않는다: (1) 변경 감지 불가 — setter를 호출해도 DB에 반영되지 않음 (Scenario A), (2) 지연 로딩 불가 — 프록시 초기화가 불가능하여 LazyInitializationException 발생 (Scenario B).

#### Q2. 변경 감지 vs merge(), 실무에서는 어떤 것을 써야 하나?
- `merge()`의 위험성: 모든 필드를 덮어씌운다는 것은 무슨 의미인가?
- 예: 이름만 바꾸려고 `merge()`를 호출했는데 다른 필드가 null로 덮어씌워진다면?
	- merge() 메서드는 엔티티가 이미 저장되어있다면 update처럼 동작하기 때문에 변경하지 않은 필드까지 덮어 씌워질 수 있다. 따라서, 필요한 필드만 update하고자 한다면 변경 감지를 이용하는 것이 낫다.

#### Q3. fetch join 말고 다른 해결 방법은 없나?
- `@EntityGraph` 활용
- DTO로 직접 조회 (`new` 키워드 사용 JPQL)
- 각각의 장단점은?
	- 강제초기화: 영속성 컨텍스트가 종료되기 전 읽고자 하는 객체를 강제 초기화. 이 방식은 서비스로직이 은연중에 뷰에 의존하게 된다. facade 계층 추가: 영속성컨텍스트의 범위를 facade service까지로 확장하면서 이 facade에서 강제초기화해 뷰에 넘기는 방식. 비즈니스로직은 각자 서비스에 두고 facade는 뷰를 위한 것들을 조합하는 느낌. layer가 하나 더 증가한다는 단점이 있음. 과거 OSIV: 해결책이긴 하나 뷰에서 엔티티를 변경해, 변경감지를 통해 DB에 반영되어버리면 문제가 된다. 스프링 osiv: 뷰에선 트랜잭션 없이 읽기만 가능하도록 해, 지연로딩 사용 가능. (가장 현대적인 해결책)
	- 피드백: 질문이 fetch join의 대안(@EntityGraph, DTO 조회)을 묻고 있는데, 답변은 주로 OSIV 관련 해결책에 집중되어 있다. 힌트로 제시된 두 가지도 언급하면 좋겠다. (1) @EntityGraph: 어노테이션 기반으로 fetch join과 동일 효과, JPQL 없이 쿼리 메서드에 붙일 수 있어 간편. (2) DTO 직접 조회: 엔티티가 아닌 필요한 데이터만 조회하므로 영속성 컨텍스트를 아예 거치지 않음.

#### Q4. OSIV=true가 편리한데 실무에서는 왜 끄는가?
- OSIV=true이면 영속성 컨텍스트가 API 응답이 끝날 때까지 살아있다 — DB 커넥션도 그때까지 반환되지 않는다면?
	- 뷰에서 엔티티를 변경할 수 있는데, 변경감지를 통해 DB에 반영되어버리면 문제가 된다.
	- 피드백: 이것은 과거 OSIV의 문제이고, 스프링 OSIV에서는 뷰에서 수정 시도 시 예외가 발생하므로 이 문제는 해결되었다. 스프링 OSIV를 끄는 핵심 이유는 커넥션 풀 고갈이다. 요청 시작~응답 완료까지 DB 커넥션을 점유하므로, 외부 API 호출이나 느린 렌더링이 있으면 커넥션 반환이 지연되어 다른 요청이 대기하게 된다.
- 트래픽이 많은 서비스에서 커넥션 풀이 고갈되면 어떤 일이 벌어지는가?
	- 뷰 레이어에서도 엔티티 조회가 가능해지는데 이러면 각 요청이 커넥션 풀을 오래 들고 있어 문제가 될 수 있지 않을까..(?)
- 컨트롤러/뷰에서 의도치 않게 지연 로딩 쿼리가 나가면 성능 문제를 추적하기 어려운 이유는?
	- 전역적으로 osiv를 true로 설정해놓았기 때문에 어떤 뷰에서 지연로딩을 사용하고 있는지 파악이 쉽지 않을 것 같다.
	- 피드백: 좋은 답변이다. 추가로, OSIV를 끄면 서비스 계층에서 의도한 쿼리만 나가도록 강제하게 되어 성능 예측 가능성이 높아진다는 점도 있다.

#### Q5. 반대로, OSIV=true가 적합한 경우는?
- 트래픽이 적은 어드민/백오피스 시스템이라면?
	- 성능을 아주 엄격하게 관리할 필요가 없다면 osiv를 열어두게 되면 lazy 읽기가 전역적으로 가능해진다. 개발 자체는 편리해지고 신규 유입 개발자들도 전체 로직 파악하게 쉬워질 것 같다.
- 빠르게 프로토타이핑해야 하는 초기 단계라면?
	- osiv = true로 설정하고 시스템이 무거워 지는 시점에 다시 고민을 할 것 같다.
- OSIV를 끄면 모든 지연 로딩을 fetch join이나 DTO로 해결해야 하는데, 그 비용 대비 효과가 항상 맞는가?
	- 뷰가 많지 않거나 성능 문제가 있지 않다면 fetch join으로 "항상 모두 가져오도록" 하는 것도 심플하게 해결할 수 있는 방안이지 않을까? AI를 잘 활용할 수 있게 된 현대에서는 DTO로 인해 코드양이 많아지거나 하는게 크게 문제가 아닌 것으로 받아들여지기 때문에 일일이 DTO를 각각 만들어주는 것 또한 해결방법일것같다.
	- 피드백: 재미있는 관점이다. 다만 DTO의 비용은 코드양보다 설계/유지보수 복잡도(어떤 DTO를 어디서 쓸지, API마다 다른 DTO 관리)에 있으므로 AI가 해결하기 어려운 부분도 있다.
