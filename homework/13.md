# 13장 과제 – 웹 애플리케이션과 영속성 관리

### 과제명
**@Transactional 범위에 따른 영속성 컨텍스트 생존 범위 이해하기**

---

### 왜 이 과제를 하는가?

실무에서 JPA를 쓸 때 가장 자주 마주치는 문제 두 가지:
1. **"save() 안 불렀는데 DB가 바뀌었다"** — `@Transactional` 안의 변경 감지를 모름
2. **"아까는 됐는데 여기서는 LazyInitializationException이 터진다"** — 트랜잭션 범위 밖의 지연 로딩을 모름

이 두 문제는 **`@Transactional`이 영속성 컨텍스트의 생존 범위를 결정한다**는 것을 이해하면 해결됩니다.
이 과제는 두 상황을 직접 체험하고, 왜 그런지 설명할 수 있게 되는 것이 목적입니다.

### 목표
- `@Transactional` 범위 안/밖에서 영속성 컨텍스트의 동작 차이를 직접 체험한다.
- 변경 감지(Dirty Checking)가 동작하는 조건을 정확히 이해한다.
- 트랜잭션 밖에서 지연 로딩 시 `LazyInitializationException`이 발생하는 이유를 이해하고 해결한다.

#### Q1. @Transactional이 영속성 컨텍스트의 생명주기를 결정한다는 것은 무슨 의미인가?
- 트랜잭션 시작 = 영속성 컨텍스트 생성
- 트랜잭션 커밋 = flush + 영속성 컨텍스트 종료
- 이 범위 밖에서 엔티티는 어떤 상태가 되는가?

**답변:**

스프링의 기본 전략(트랜잭션 범위의 영속성 컨텍스트)에서는 `@Transactional`이 영속성 컨텍스트의 생사를 결정한다.

```
@Transactional 시작
  → EntityManager 생성 → 영속성 컨텍스트 생성
  → 엔티티 조회/수정 (영속 상태 — 1차 캐시, 변경 감지 동작)
@Transactional 종료
  → flush (변경 감지 → UPDATE SQL)
  → 트랜잭션 커밋
  → 영속성 컨텍스트 종료
  → 엔티티는 **준영속(detached) 상태**가 됨
```

준영속 상태의 엔티티는:
- 1차 캐시에서 제거됨 → `findById`로 다시 조회하면 새로운 SELECT 발생
- **변경 감지가 동작하지 않음** → setter를 호출해도 DB에 반영 안 됨 (Scenario A-2)
- **지연 로딩 불가** → 프록시 초기화 시 `LazyInitializationException` 발생 (Scenario B-1)

이것이 `changeName`(트랜잭션 O)은 UPDATE가 반영되고, `changeNameWithoutTransaction`(트랜잭션 X)은 반영되지 않는 이유다.

#### Q2. 변경 감지 vs merge(), 실무에서는 어떤 것을 써야 하나?
- `merge()`의 위험성: 모든 필드를 덮어씌운다는 것은 무슨 의미인가?
- 예: 이름만 바꾸려고 `merge()`를 호출했는데 다른 필드가 null로 덮어씌워진다면?

**답변:**

실무에서는 **변경 감지(Dirty Checking)를 사용해야 한다.** `merge()`는 가급적 피한다.

`merge()`의 동작 방식:
```java
// 준영속 엔티티를 merge
Member merged = em.merge(detachedMember);
// → DB에서 id로 조회 (SELECT)
// → detachedMember의 **모든 필드**를 조회된 엔티티에 덮어씌움
// → 트랜잭션 커밋 시 UPDATE
```

문제는 "모든 필드를 덮어씌운다"는 점이다:
```java
Member detached = new Member();
detached.setId(1L);
detached.setName("newName");
// age = 0 (기본값), team = null (설정 안 함)

em.merge(detached);
// → name = "newName", age = 0, team = null 로 UPDATE!
// → 기존에 있던 age=20, team=teamA 가 사라짐
```

반면 변경 감지는:
```java
@Transactional
void changeName(Long id, String newName) {
    Member member = memberRepository.findById(id).orElseThrow();
    member.changeName(newName);  // name만 변경
    // → UPDATE member SET name=? WHERE id=?  (name 필드만 변경됨)
}
```

변경 감지는 **변경된 필드만** UPDATE하므로 의도하지 않은 데이터 손실이 없다.

#### Q3. fetch join 말고 다른 해결 방법은 없나?
- `@EntityGraph` 활용
- DTO로 직접 조회 (`new` 키워드 사용 JPQL)
- 각각의 장단점은?

**답변:**

LazyInitializationException을 해결하는 방법은 크게 3가지:

**1) fetch join (JPQL)**
```java
@Query("select m from Member m join fetch m.team where m.id = :id")
Optional<Member> findWithTeamById(@Param("id") Long id);
```
- 장점: 한 번의 쿼리로 연관 엔티티까지 조회, 명시적
- 단점: JPQL을 직접 작성해야 함, 페이징과 함께 사용 시 주의 필요 (컬렉션 fetch join은 페이징 불가)

**2) @EntityGraph**
```java
@EntityGraph(attributePaths = {"team"})
Optional<Member> findById(Long id);
```
- 장점: JPQL 없이 어노테이션만으로 fetch join 효과, 간결함
- 단점: 복잡한 조건이나 다중 연관관계에서는 한계, LEFT OUTER JOIN으로 동작

**3) DTO 직접 조회**
```java
@Query("select new homework.dto.MemberDto(m.name, t.name) from Member m join m.team t where m.id = :id")
MemberDto findMemberDtoById(@Param("id") Long id);
```
- 장점: 필요한 컬럼만 조회 → 성능 최적화, 엔티티가 아니므로 영속성 컨텍스트 무관
- 단점: DTO 클래스를 별도로 만들어야 함, 엔티티가 아니므로 변경 감지 불가

단, DTO 직접 조회를 Repository에 바로 넣으면 **계층 책임이 섞이는 문제**가 있다. Repository는 엔티티를 다루는 계층인데 DTO(프레젠테이션 객체)를 반환하면 관심사가 혼재된다. DTO 구조가 바뀔 때마다 Repository를 수정해야 하는 문제도 생긴다.

실무에서는 **커맨드/쿼리 Repository를 분리**하는 패턴으로 해결한다:

```java
// 엔티티용 — 순수 Repository (변경 담당)
public interface MemberRepository extends JpaRepository<Member, Long> { }

// 조회 전용 — DTO 반환 전담 (조회 담당)
@Repository
public class MemberQueryRepository {
    private final EntityManager em;

    public List<MemberDto> findMemberDtos() {
        return em.createQuery(
            "select new MemberDto(m.name, t.name) from Member m join m.team t",
            MemberDto.class
        ).getResultList();
    }
}
```

이렇게 분리하면 `MemberRepository`는 엔티티만, `MemberQueryRepository`는 DTO 조회만 담당하여 각 계층의 책임이 명확해진다.

실무 가이드:
- 단순 연관관계 로딩 → **@EntityGraph** (가장 간결)
- 복잡한 JOIN 조건 → **fetch join** (명시적 제어)
- 조회 전용 API → **DTO 직접 조회** (QueryRepository 분리 권장)
